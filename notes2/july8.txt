LPS / Spiking Network Debugging Summary + Implementation Prompt
Based on the July 8 transcript and LPS (9) notes

MAIN POINT
The next implementation goal is not only to change the learning equations. The bigger goal is to make the simulation observable enough that we can tell whether the winner-take-all behavior is actually happening for the right physical reasons.

The transcript says to build two required debugging plots:

1. Raster plot
   - Shows every spike from every neuron over time.
   - Y-axis: every node/neuron in the network.
   - X-axis: simulation time steps.
   - A mark/dot/tick appears whenever a node spikes.
   - This is the first debugging tool because it shows whether the network activity pattern matches the expected winner-take-all story.

2. Accumulated charge / ACCQ heatmap over time
   - This is the “other plot” discussed in the transcript.
   - It is similar to a raster in shape: nodes vertically, time steps horizontally.
   - Instead of only marking spikes, each cell is colored by the neuron’s internal accumulated charge / voltage / ACCQ at that time step.
   - Cold/blue means low charge.
   - Hot/yellow/red means close to threshold.
   - After a spike, the value should drop/reset to cold.
   - After inhibition arrives, connected E neurons should drop to colder values.
   - This plot shows what the raster cannot show: how each neuron is building charge, getting inhibited, resetting, or staying stuck.

The transcript also mentions that a weight heatmap can be useful later, but the two required plots right now are the raster plot and the ACCQ/sum heatmap.


EXPECTED NETWORK BEHAVIOR TO CHECK
When the simulation is brand new:

1. The input/sensory pattern should fire first.
   Example: if the pattern has three active sensory cells, those three sensory nodes should spike.

2. E neurons should integrate charge over multiple steps.
   They should not all instantly fire unless the input layer has been forced/maxed so strongly that it drives them immediately.

3. At first, before inhibition has learned/settled, E firing may look like “popcorn.”
   Several E neurons may fire close together because their initial random weights are similar and nothing has suppressed the pool yet.

4. Eventually I1 should fire after enough E activity reaches it.
   I1 may appear on the same step as an E spike if the simulator evaluates layers in the same step, but conceptually it is caused by prior/current E activity.

5. After I1 fires, connected E neurons should quiet down.
   In the raster, fewer E neurons should spike after inhibition.
   In the ACCQ heatmap, connected E neurons should get colder / lose accumulated charge after inhibition.

6. If I1 fires but E accumulated charges do not drop, that likely means the topology or connection logic is wrong.
   The visualization may show a connection, but the actual network may not have that physical edge.

7. If a neuron gets hot/yellow and stays hot without spiking or resetting, that is a problem.
   It may mean threshold crossing/reset logic is wrong.

8. If a neuron stays cold/blue forever, that is also a clue.
   It may mean the neuron is disconnected, receiving no input, always inhibited, or prevented from integrating.

Data needed:
- node order: stable list of node ids
- spike events: {time, node_id, node_type}

Basic pseudocode:

for each time step t:
    run simulation update
    for each node:
        if node.spiked:
            spike_events.append({time: t, node_id: node.id, type: node.type})

Plot:
- x = time
- y = node index / node id
- mark a dot, tick, or small square when spike occurs
- optionally use different colors/shapes for sensory, E, and I nodes
- include all nodes, not only winners

What to inspect:
- Do active sensory nodes spike when a pattern is presented?
- Do E neurons take time to integrate before firing?
- Does I1 fire after E activity?
- After I1 fires, do other E neurons stop/quiet down?
- Are there unexpected neurons firing that should not fire?
- Are there missing spikes from nodes that should be active?


ACCQ / SUM HEATMAP IMPLEMENTATION
Goal: show what is happening inside each node over time.

Data needed:
- charge_matrix[node_index][time] = accumulated charge / voltage / ACCQ
- optional spike overlay
- optional inhibition-event overlay

Basic pseudocode:

for each time step t:
    run simulation update
    for each node index i, node in nodes:
        charge_matrix[i][t] = node.accq_or_voltage_value_at_consistent_stage
        spike_matrix[i][t] = 1 if node.spiked else 0
        inhibition_matrix[i][t] = amount_of_inhibition_received

Plot:
- x = time steps
- y = node ids
- color = accumulated charge / threshold-normalized charge
- cold color = low charge
- hot color = near threshold
- after spike/reset = should drop cold
- after inhibition = should drop colder

Recommended normalization:
- normalized_charge = node.accq / node.threshold
- threshold is around 1.0 if that is your base setting
- values above 1.0 mean the neuron crossed threshold
- allow negative values if inhibition can hyperpolarize below rest
- do not clamp at zero if the newer model allows hyperpolarization

Useful overlays:
- small dot or black marker when spike happens
- vertical line or marker when I1 fires
- markers for inhibition arrival
- trial boundary markers

What to inspect:
- Do E neurons gradually warm up before spiking?
- Does the winner hit threshold first?
- Do losing E neurons cool down after I1 inhibition?
- Is any neuron staying hot without firing/resetting?
- Is any neuron staying cold even though topology says it should receive input?

FREE ENERGY / CHARGE MATH FROM THE TRANSCRIPT
The transcript explains free energy as threshold minus instantaneous accumulated charge.

Plain version:
FE = threshold - IAccQ
IAccQ = sum(weight_i * input_i)

Meaning:
- IAccQ is the charge received from the current input event(s).
- If IAccQ reaches or exceeds threshold, the neuron can fire immediately.
- If IAccQ is below threshold, the remaining gap is free energy / unexplained amount.
- When the gap is small, the neuron is close to firing.
- When the gap is large, the neuron still needs more charge.

Important implementation detail:
- Use the node’s threshold, even if all nodes currently share the same threshold from the base node class.
- The transcript says this is “both” global and individual: each node has its own threshold variable, but right now the same base threshold may be used for all pyramidal/excitatory neurons.

Possible code:

IAccQ = sum(edge.weight * source.output_signal for edge in incoming_edges)
FE = node.threshold - IAccQ

if IAccQ >= node.threshold:
    node.spike()


LEARNING EQUATION STATUS
Do not let the implementation silently choose the wrong equation. Ask for confirmation.

From the board/transcript, the update form appears to use:
- learning rate
- free energy term
- saturation term that slows learning near max weight
- input/spike signal, where +1 means spike and -1 means no spike / decay case

Candidate form from the board:
Delta w = LR * FE * (1 - (w / w_max)^2) * input_signal

Previous notes/conversations may also reference this similar but different form:
Delta w = LR * FE * (1 - w / w_max)^2 * learn_signal

These are not the same. They behave differently near high weights.
The current transcript/whiteboard seems to emphasize the w-ratio squared inside the term: 1 - (w / w_max)^2. But this should be asked/confirmed before implementing as final.

Also ask whether potentiation and depression should use the same learning rate.
The transcript says the current equation assumes weights increase and decrease at the same rate, but biology may have slower decay/depression than learning/potentiation. So the implementation should support separate rates even if defaults are equal.

Suggested flexible implementation:

if input_signal == +1:
    lr = LR_plus
else if input_signal == -1:
    lr = LR_minus
else:
    lr = 0

sat = 1 - (w / w_max)^2      # if confirmed
# alternative: sat = (1 - w / w_max)^2

delta_w = lr * FE * sat * input_signal
w = clamp(w + delta_w, w_min, w_max)


PRETRAINED / MAXED SENSORY WEIGHTS QUESTION
This is one of the biggest unclear points in the transcript.

What the transcript says:
- One idea was to start the input/sensory-to-next-layer weights at max so the presented pattern immediately drives the next layer.
- The reason for doing that was to force the pattern into the second layer so the model can focus on downstream competition.
- But there is a concern: if these input weights are maxed, the receiving neurons start with very high charge.
- Then inhibitory learning can happen too strongly/too quickly, possibly in one shot.
- That can mess up competition because everything may quickly start representing everything.
- A more natural approach may be to start low/random and give the system more time steps so it learns layer by layer.

Implementation recommendation:
Do not hard-code one answer yet. Implement this as a switchable initialization mode:

input_init_mode = "max_pass_through" or "random_learned"
freeze_input_layer = true or false
inhibition_learning_warmup = number of trials before I1 plasticity starts

Modes to test:

Mode A: max_pass_through
- S/receptor/input weights start near w_max.
- This makes the sensory pattern immediately visible downstream.
- Risk: may cause fast inhibitory overlearning.

Mode B: random_learned
- S/receptor/input weights start random/low.
- Network gets more time steps/trials to self-organize.
- More biologically natural, but slower.

Mode C: staged training
- First allow sensory/input layer to become reliable.
- Then turn on stronger inhibitory plasticity.
- This may avoid one-shot inhibitory saturation.

The raster and ACCQ heatmap should be used to compare these modes.
